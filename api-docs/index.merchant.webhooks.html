Webhooks Integration

Real-time subscription status notifications for seamless integration

Overview

LunarPay uses webhooks to notify your application about important subscription status changes in real-time. This ensures your system stays synchronized with subscription lifecycle events across all scenarios.

How Webhooks Work

Configure your webhook endpoints from the dashboard under Settings > Integrations. Events will be delivered via HTTP POST to your specified URLs. Your listener must authenticate requests using a Bearer token, which can also be set and retrieved from the dashboard.

Supported Events

  • subscription_created - New subscription created
  • subscription_updated - Subscription status or details updated

Webhook Trigger Scenarios

Webhooks are sent in two different scenarios:

1. Manual Events (Immediate)

  • When: User-initiated actions occur
  • Examples: Customer creates a subscription, cancels a subscription, or a payment attempt happens (success or failure)
  • Event Types: subscription_created, subscription_updated
  • Note: Webhooks are only sent if a status field changes as a result of the action.

2. Time-Based Status Changes

  • When: Time-based events automatically change subscription status
  • Examples: Trial period ends, a scheduled payment succeeds or fails, the 2-day grace period expires, the access period ends.
  • Event Type: subscription_updated

When Webhooks Are Sent

LunarPay only sends a webhook when at least one of the following key status fields has changed since the last webhook was sent:

  • c_status – Overall subscription status
  • trial_status – Trial period status
  • access_period_status – Access window status
  • last_transaction_id – Most recent transaction ID (successful or failed)

Changes to other fields (e.g., plan_type, amount, next_payment_on) will not trigger a webhook unless one of the above status fields changes as well.

Status Field Explanations

Customer Status (c_status)

  • on_trial – Subscription is in trial period with no successful payments yet
  • active – Subscription is active and payments are current
  • unpaid – Subscription payment is overdue or failed
  • cancelled – Subscription has been cancelled

Trial Status (trial_status)

  • active – Trial period is currently active (customer can access services without payment)
  • ended – Trial period has ended (either naturally or due to early payment)
  • null – Subscription has no trial period

Access Period Status (access_period_status)

  • active – Customer has access to services (even if c_status is unpaid or cancelled)
  • ended – Access period has ended (customer no longer has access)
  • null – No access period set

Last Transaction ID (last_transaction_id)

The ID of the most recent transaction. This field updates when a new payment is processed (successful or failed).

Example Status Progression

  • c_status: "on_trial", trial_status: "active", access_period_status: "active"
  • c_status: "active", trial_status: "ended", access_period_status: "active"
  • c_status: "unpaid", access_period_status: "active"

Implementing Webhook Listeners

Authentication

All webhook requests include a Bearer token in the Authorization header. Validate this against your dashboard config.

Response Requirements

  • Return 200 OK to confirm receipt
  • Process data within 10 seconds

Example Webhook Handler (Node.js)

app.post('/webhooks/subscription', (req, res) => {
  const token = req.headers.authorization?.replace('Bearer ', '');
  if (token !== YOUR_WEBHOOK_TOKEN) {
    return res.status(401).json({ error: 'Unauthorized' });
  }

  const { event_type, data } = req.body;

  switch (event_type) {
    case 'subscription_created':
      handleSubscriptionCreated(data);
      break;
    case 'subscription_updated':
      handleSubscriptionUpdated(data);
      break;
  }

  res.status(200).json({ received: true });
});

  const { c_status } = data;

    // Disable access
  } else if (c_status === 'active') {
    // Enable access
  }
}

🔧 Troubleshooting

  • Webhook not received: Check endpoint URL and network access
  • Auth error: Verify Bearer token
  • Status not updating: Use c_status, not status
